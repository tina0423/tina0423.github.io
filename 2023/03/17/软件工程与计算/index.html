软件工程与计算

软件工程的发展

1970 - 瀑布模型的提出

  • 软件的需求变化改变大，文档不易改变

  • 硬件按摩尔定律发展，软件跟不上硬件的变化

  • 越早发现错误（在需求阶段），所付出的代价越低，越晚发现错误，代价越高

1980 - 效率，面向对象，软件过程模型

硬件发展

  • 个人电脑的推广
  • GUI编程环境
  • CD-ROM光盘

应用

  • 软件复杂度的指数级增长
  • 个人消费市场的增加
  • 日本的流水线模式带来的高效率

软件开发系统化

  • 软件结构分析设计
  • 面向对象的分析设计
  • 软件重用思想
    • 数学库->抽象数据类型 -> C++ ->模型驱动的开发

过程模型&过程评价&使用工具支持过程

包括原型->渐进交付->演化式开发->螺旋

CMU SEI 建立了软件能力成熟度模型（SW-CMM)，通过评价企业的开发过程管理来反映企业的生产能力

1990 - 局域网，软件架构，OO

  • 整合以前建立的各个“应用孤岛”，实现对整个企业业务的全面整体化管理
  • 以局域网为基础的软件系统可以覆盖企业的所有重要部门、重要人员和重要工作
  • 过程模型
    • RUP
    • Agile
    • 产品线
  • 开源软件

2000 - 互联网，Agile

  • 不断涌现出来的需求 <-> 新兴的需求 需求是在不断接触的过程中出现的

2010 - 云原生

微服务、云原生

continuous software engineering

serverless

前端开发基础

浏览器架构

  • 用户界面 -> 浏览器引擎 -> 渲染引擎 -> (网络/JS解释器/)
  • 用户界面：负责于用户的交互，如地址栏输入、刷新、前进与后退

超文本语言

  • HTML解析树
  • CSS层叠样式表，用于控制网页的样式和布局；HTML负责定义网页里有哪些内容，CSS决定内容是以何种形式（字体、颜色）展现，CSS必须和HTML同时出现
  • 渲染树将CSS和HTML一起渲染之后调用系统绘制，显示出来

JavaScript

  • 让网页“动起来”，通过JavaScript实现用户和服务器的交互

  • 可以把JS直接放入HTML，也可以放在一个文件里

  • Async关键字：使脚本执行和DOM构造异步化

Vue.js

  • 声明式的渲染
  • 组件化应用构建
  • 前端路由：更新页面的过程不需要把浏览器所有页面重新渲染，而是以单页面为基本单元，只渲染改变的部分
  • 强大的构建工具

Node.js

后端开发基础

  • 协议栈

    TCP/IP的四层协议栈

  • Tomcat：和HTTP服务器一样，绑定IP并监听TCP端口；管理SERVLET程序的生命周期

  • REST风格：可重复、分层、无状态、可靠、统一接口、可扩展、可重用

    前后端分离：只返回数据，不返回由界面任何有关内容

  • JSON：本质是正则表达式，可以被有限状态机解读

  • Spring Boot

  • 数据设计

  • JDBC原理(java database)

DAO

  • DAO：数据存取对象，位于业务逻辑层和数据层之间实现以面向对象API为形式的对持久化数据进行CRUD操作的对象

接口是根据业务需要制定，不需要全部开放

项目管理

项目

  • 具有一个明确目标
  • 由限定的开始和结束日期
  • 由成本限制
  • 消耗人力和非人力资源
  • 多工种合作

项目管理的目标

  • 在限定时间内
  • 在一定的成本内
  • 在要求的质量水平上
  • 高效使用资源
  • 获得客户认可

过程组与活动

  • 过程组：项目启动、项目计划、项目执行、项目跟踪与控制和项目收尾
  • 活动：计划制定、团队管理、成本控制、质量保障、度量、过程管理、进度跟踪与控制、风险管理（员工跳槽；流感；项目经理跳槽等，需要及时为这些风险做好相应的准备）、配置管理

团队

  • 软件项目依赖于有效的团队组织和管理
  • 团队：为了一致目的、绩效标准、方法而共担责任并且技能互补的少数人
  • 主程序员团队
  • 民主团队
  • 开放团队

避免以下情况

  • 地理分散&时间分割
  • 产品质量的降低（代码质量差）：代码重构&code review
  • 虚假的最后期限
  • 小圈子控制：尽量客观做事

软件质量保障（QA）

  • 因素：
    • 产品运维：可用性，可靠性，高效
    • product revision: reusability maintainability portable

评审过程

规划 –> 总体部署 –> 准备 –> 审查会议 –> 返工 –> 跟踪

规划：制定审查计划，决定审查会议次数，安排时间和地点，参与人员，审查内容

总体部署：想参与会议人员描述审查材料内容，审查目标和一些假设，并分发文档

准备：审查人员各自独立执行检查内容（会议上不会进行讨论，而是展示）

审查会议：通过讨论，识别、确认、分类发现的错误（不要开成批斗大会）

返工：修改发现的缺陷

跟踪：确认所有发现的问题都得到了解决

质量度量

你不能控制自己无法度量的东西

软件配置管理

中间制品：除了最终产品外，软件开发中还会产生中间制品，包括文档，需求分析模型，软件体系结构设计模型等。这些制品是不同阶段、不同角色、不同软件开发活动进行协同的基础

在复杂系统中，产生的制品数量众多，开发者需要维护一个清单才能清楚项目所处的状态，理解已经完成的工作和将要进行的工作

配置管理：用技术和管理的指导和监督方法，来标识和说明配置项的功能和物理特征

配置管理活动：

标识配置项，版本管理，变更控制，配置审计，状态报告，软件发布管理

分支管理策略

  • 主分支
  • 开发分支
  • 临时性分支

管理实践

软件项目的投入

  • 人员的成本
  • 工具的购买（开发工具，办公套件）
  • 培训的费用：开发人员接受培训，获得开发项目所需技能的费用
  • 差旅费
  • 维护费用
  • 生产停顿的损失
  • 市场和服务的费用：推广费、参展费
  • 机会成本

软件项目的产出

  • 节约商业活动成本：例，开发了新的库存管理系统后，加快了流通并减少了浪费
  • 创新商机增加销售
  • 提高品牌含金量

”三驾马车“

  • 产品经理：精准定位用户的需求
  • 项目经理
  • 技术经理

目标驱动

SMART原则：

  • specific
  • measurable
  • achievable
  • time

给管理新手的建议

  • 不要将流程与目标相混淆

需求分析

需求工程

  • 概念：所有需求处理活动的总和。收集信息，分析问题，整合观点，记录需求并验证正确性

  • 三个主要任务

    • 明确软件的应用环境及其目标，做什么和为什么
    • 将目标和功能反映到软件系统中，映射为可行的软件行为，并对软件行为进行准确的规格说明（specification）
    • 随着时间外界的变化及时调整目标

需求开发过程模型

  1. 需求获取（从利害相关人中获取）
  2. 需求分析
  3. 需求规格说明
  4. 需求验证

1.需求获取

从人、文档或环境获取需求

常见困难
  • 用户和开发人员背景不同，立场不同：“床边B超，肝胆胰脾”
  • 普通用户缺乏概括性、综合性的表述能力：”不聪明的记者“
  • 用户存在认知困境：”平板电脑的概念不理解“
  • 用户越俎代庖：对名词不理解“双机热备”
  • 缺乏用户参与
